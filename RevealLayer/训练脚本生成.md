# RevealLayer 训练脚本生成

## 目标

在 `E:\Image_Decomposition\code\RevealLayer` 中，基于现有推理代码和 `RevealLayer.pdf` 复现一套可训练的 LoRA 训练脚本。训练实现必须尽量贴近论文描述，同时尊重仓库已有模型、pipeline、数据格式和推理路径。

当前已知状态：

- 仓库是 RevealLayer 论文的代码实现，但目前主要包含推理脚本。
- 论文提到训练细节，但仓库中不一定包含完整训练所需模块。
- 推理入口主要围绕 `infer.py`、`infer_z_order.py`、`models/custom_pipeline.py`、`models/custom_model_mmdit.py`、`models/custom_model_xvae.py`、`models/adapter.py` 和 `benchmark/RevealLayerBench.json`。

## 工作原则

- 先读代码和论文，再写训练代码。不要只凭 Diffusers/Flux 通用训练模板套壳。
- 以推理路径为锚点：训练数据、bbox/query 组织、layer 数量、latent shape、prompt、VAE/transformer/adapter 调用方式要和推理脚本能互相解释。
- 如果论文细节、权重结构或代码实现之间有冲突，优先记录差异，并实现最小可运行且可解释的版本。
- 不要臆造论文没有明确支持的 loss 或模块。缺失信息要在训练文档中标注为“工程近似”或“待论文/作者代码确认”。
- 默认只生成脚本、配置和说明文档；除非用户明确要求，不要启动真实训练。
- 保持改动集中，不重构无关推理代码。必要的共享逻辑可以抽成小型 helper，但不要大规模改目录结构。

## 必须先检查的内容

开始实现前，至少检查这些文件：

- `infer.py`：推理入口、参数、数据读取、bbox/query 构造、pipeline 调用。
- `infer_z_order.py`：如果训练目标涉及层级顺序、合成或 overlap 样本，检查其图层合成顺序和输入 JSON 约定。
- `models/custom_pipeline.py`：模型组件如何拼装、latent/image_ids/prompt embeds 如何传递。
- `models/custom_model_mmdit.py`：可训练主体、attention/adapter/LoRA 可插入位置。
- `models/custom_model_xvae.py`：RGBA/透明图层解码相关接口。
- `models/adapter.py`：论文中额外模块或 occlusion/layer adapter 的代码对应。
- `configs/*.py`：分辨率、模型路径、推理默认配置。
- `requirements.txt`：训练所需依赖是否已存在，例如 `accelerate`、`peft`、`tensorboard`、`safetensors`。
- `RevealLayer.pdf`：训练目标、loss、数据、冻结/微调范围、LoRA 设置、优化器和 batch 设置。

如果需要从 PDF 中抽取信息，优先提取训练相关章节、公式、实验设置、附录和 ablation，不要泛读后直接概括。

## 实现流程

1. 建立论文到代码的映射表。
   - 记录论文中的每个训练模块、loss、数据字段和模型组件。
   - 标明它在当前代码中的对应文件/函数。
   - 对找不到实现的部分，写清楚需要新增、近似实现还是暂不实现。

2. 设计数据集接口。
   - 数据加载必须和推理脚本的输入语义对齐。
   - 支持 manifest/JSONL/JSON 形式，字段至少覆盖原图、prompt、bbox 列表、层数、目标 RGBA layers、背景或合成图。
   - bbox 坐标、图像路径、相对路径根目录、RGBA 通道、空层、缺失层都要有明确处理。
   - Windows/PowerShell 导出的 JSON 可能带 BOM，读取 JSON 时优先使用 `encoding="utf-8-sig"`。

3. 设计训练前向。
   - 复用推理 pipeline 中的模型加载、图像预处理、bbox/query 编码、latent/image_ids 构造。
   - 冻结基础模型，只打开论文指定模块或 LoRA 参数。
   - LoRA 默认接入 transformer/MM-DiT 中的 attention/linear 模块；具体 target modules 必须从当前模型命名中自动发现或显式校验。
   - 保证保存的 LoRA 权重能被后续推理脚本加载，必要时同步新增推理加载参数。

4. 实现 loss。
   - 先实现论文明确描述的 loss。
   - 扩散训练 loss 要和当前 scheduler / prediction type 对齐。
   - 如果论文包含 RGBA、alpha、mask、composition、occlusion、layer consistency 或 z-order 相关监督，逐项实现并提供权重参数。
   - 对工程近似 loss，必须在代码注释和训练文档中说明原因。

5. 加入训练基础设施。
   - 使用 `accelerate` 支持单卡/多卡、混合精度、梯度累积、断点恢复。
   - 支持 TensorBoard loss log。
   - 支持 checkpoint 保存、最新 checkpoint 恢复、指定 checkpoint 恢复。
   - 支持 `gradient_checkpointing`、`xformers` 或 PyTorch SDPA 等训练加速开关，按环境可用性降级。
   - 支持 seed、数据加载 worker 数、batch size、学习率、warmup、训练步数、保存间隔、验证间隔等 CLI 参数。

6. 补充文档。
   - 写清楚数据格式、路径布局、启动命令、多卡命令、恢复训练命令、日志位置、权重保存位置。
   - 写清楚哪些训练细节来自论文，哪些来自当前代码推断，哪些是工程近似。
   - 给出最小 smoke test 命令，但不要默认运行长时间训练。

## 推荐产物

优先生成这些文件，具体路径可按仓库现有风格微调：

- `train_reveallayer_lora.py`：主训练入口。
- `training/dataset.py`：RevealLayer 数据集和 collate 逻辑。
- `training/losses.py`：训练 loss 组件。
- `training/lora.py`：LoRA 注入、可训练参数选择、保存/加载 helper。
- `configs/train_reveallayer_lora.yaml`：默认训练配置。
- `docs/train_reveallayer_lora.md` 或更新 `Reveallayer.md`：训练说明文档。

如果仓库已有类似目录或命名，优先沿用现有风格。

## 验收标准

完成后至少验证：

- `python -m py_compile` 能通过新增/修改的 Python 文件。
- 数据集能用一个小 manifest 初始化并返回 batch，shape、dtype、路径解析和 bbox 数量符合预期。
- LoRA 注入后只有预期参数 `requires_grad=True`。
- 训练脚本支持 `--help` 并展示关键参数。
- checkpoint、TensorBoard log、输出目录参数完整。
- 文档中的命令和脚本参数一致。

如用户明确要求 smoke test，可以只跑极小步数，例如 1 batch / 1 step，并避免触发完整训练。

## 需要特别报告的风险

- 论文训练细节缺失，导致只能实现近似版本。
- 当前仓库缺少训练所需的目标 RGBA layer 数据或 metadata。
- 模型自定义组件和标准 Diffusers/PEFT LoRA 接口不完全兼容。
- 推理权重保存格式与训练 LoRA 权重格式需要额外适配。
- 训练显存需求过高，需要降低分辨率、batch size、layer 数或启用 checkpointing。
